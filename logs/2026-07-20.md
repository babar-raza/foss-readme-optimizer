- [master] **2026-07-20 — Decision 40: mode-gate split (read-only vs. push access), the
  `aspose-page-foss` profiling latency actually fixed (not just re-documented), and a `GOV-014`
  compliance sweep.** User asked how large-repo profiling latency should be handled at scale on
  free GitHub-hosted runners, given it might recur across several huge registry repos. Investigation
  (three parallel Explore agents plus direct reading) found the aspose-page-foss cost had been
  independently re-narrated as "documented, open, not fixed" twice (Wave 3, Wave 6) without ever
  getting a `BACKLOG` row, and that the framing of it as low-urgency rested on a misreading:
  `mode: "disabled"` was blocking every read-only capability from ever running against it, when it
  was only ever meant to gate push/full-cycle access. User confirmed both: fix the access-model
  conflation, and make `GOV-014` compliance actually stick this time.

  **Part A — mode-gate split.** Traced `entry.mode == "disabled"` as an independently duplicated
  check across 7 call sites, blocking reads and writes identically:
  `capabilities/profile_repository.py`, `get_product_facts.py`, `detect_readme_gaps.py`,
  `classify_upstream_change.py` (all read-only), `orchestrator.inspect_repo()` (read-only),
  `orchestrator.require_permitted()` (genuinely mutating: `generate_repo()`/`run_repo()`'s
  render+commit path), and `supervisor/loop.py::supervise_repo()`'s entry gate (mixed). New
  `registry/loader.py::require_listed(org_repo)` — raises `NotAllowlistedError` only if the entry
  isn't in `data/products.json` at all, mode irrelevant — now gates every read-only capability and
  `supervise_repo()`'s entry point; `require_permitted()`/`is_permitted()`/`enabled_entries()` are
  unchanged and still strictly gate the mutating pipeline. Safety-critical companion, since relaxing
  `supervise_repo()`'s entry gate removed the *only* place mode was ever checked before a
  write-capable dispatch: `supervisor/loop.py::_dispatch_and_record()` now independently refuses to
  dispatch any `local_write`/`remote_write` capability unless `entry.mode == "full"` (proven by a
  focused unit test with a fake write-capable manifest, `TestWriteCapableModeGate` — no real
  write-capable capability is registered yet to exercise this live, flagged not hidden, same as
  every other `PARTIAL` claim in this project). Decision #40 recorded; `CORE-004` revised (previously
  read "disabled or unlisted... blocked... no clone", corrected to unlisted-always-blocked,
  disabled-blocks-writes-only); new `CORE-033`; `OPS-007` revised to name both gates. Docs updated to
  match: `AGENTS.md`'s allow-list section and "equal precedence for research" bullet, `docs/
  safety-model.md`'s allow-list section. Explicitly out of scope, flagged not silently decided:
  `run_repo()`/`run_registry()` still skip disabled entries entirely (unchanged) — making the full
  render/commit pipeline perform an analysis-only pass for a disabled entry is a further, separable
  change to its internal branching, not this one.

  **Part B — the profiling latency, actually fixed.** Root cause read directly from the code:
  `clone_baseline()` never caches (always destroys + re-clones); `_find_manifest_paths()`
  (`inspection/file_inventory.py`) walks the whole tree whenever most of the 6 registered
  ecosystems' patterns never resolve, true of any single-ecosystem repo. The clone-vs-walk split
  was never measured before this pass — both survey scripts (`survey_full_registry_ecosystem_
  detection.py`, `survey_full_registry_wave6_reconciliation.py`) timed them combined. Instrumented
  both to report `clone_s`/`profile_s` separately, then live-verified against the real repo
  (`scripts/retrofits/verify_profile_cache_against_aspose_page_foss.py`, kept after use):
  **`clone_s=199.2s`, `profile_s=0.0s`** — the walk was never the bottleneck; 100% of the cost is
  the git clone (network transfer of a ~1GB repo). This directly overturned the plan's own initial
  assumption (that the walk was the primary lever) before any of the actual fix was built — instrumentation-
  before-optimizing paid off immediately. Built anyway, as a circuit breaker rather than the
  fix it turned out not to need to be: `file_inventory.py::_MAX_FILES_SCANNED` (200,000) bounds
  `_find_manifest_paths()`'s walk against a genuinely pathological tree, documented as an honest
  precision/latency trade-off in the same style the file already uses. The real fix, matching the
  live numbers: a pre-clone freshness gate. New `gitsafety/clone.py::remote_head_sha()` (`git
  ls-remote`, no clone, one tiny round-trip); new `state/schema.py::ProfileCacheV1`, its own field
  on `RunStateV1` (deliberately not reusing `upstream_revision_at_accept`/`supervisor_state` — the
  same multi-writer-clobber bug decisions #32/#34 already fixed, one level up); new `profile/
  cached.py::get_or_build_profile()`, called by both `profile_repository.execute()` and
  `get_product_facts.execute()` in place of their previously-duplicated inline
  `clone_baseline`+`build_profile` pair — cache hit returns the stored profile with zero clone.
  Live-proven in the same verification run: **the second call, via `get_or_build_profile()`, took
  0.97s** against the identical real repo — a ~205x reduction for the unchanged-upstream case, which
  is the realistic shape of a scheduled/repeat registry sweep. Explicitly rejected: reusing the
  existing `capability_outputs`/`idempotency_key` ledger (`effect_ledger.py`) — verified by reading
  it, that fingerprint hashes declared input *arguments* only (`org_repo`), which would cache forever
  regardless of upstream changes; it solves EFF-002's replay-avoidance problem, not this one, correctly
  not conflated.

  Investigated, and deliberately *not* built: a size probe + `git clone --filter=blob:none` for the
  remaining cold-clone case. Tested directly (against the live repo, and a local `file://`-transport
  spike) and confirmed this would not help: `clone_baseline()` always performs a full working-tree
  checkout, and a normal checkout after a blobless partial clone still fetches every blob it needs
  regardless of the filter — the bandwidth saving only materializes paired with `--no-checkout`/`git
  sparse-checkout`, which would mean replacing `_find_manifest_paths()`'s `os.walk()`-based scan with
  a git-tree enumeration entirely (a real architecture change, not a flag). Logged as `BACKLOG` row
  `SCL-004` rather than half-implemented for a benefit it wouldn't actually deliver. Registry-scale
  hygiene: `run_registry()` now cleans up each entry's baseline clone in a `finally` once that
  entry's run is done (`gitsafety/clone.py`'s `_force_rmtree` promoted to public `force_rmtree` and
  reused here rather than a fourth duplicate of the same function) -- a multi-repo batch no longer
  accumulates every processed repo's baseline clone on disk for the rest of the job. A separate
  wall-clock timeout wrapping `build_profile()` itself was considered and skipped: `_MAX_FILES_
  SCANNED` already bounds the actual cost driver, and a true wall-clock wrapper around a synchronous
  `os.walk()` needs a thread/subprocess boundary (`signal.alarm` isn't available on Windows, this
  project's own dev platform) — disproportionate complexity for a cost already bounded another way.

  **Part C — `GOV-014` compliance.** The rule itself already read as a hard "MUST"
  (`plans/requirements.md`) — the problem was compliance, not wording, evidenced by the
  `aspose-page-foss` finding being independently re-narrated twice without a row. Strengthened with
  an explicit mechanical trigger: narrative phrasing ("documented, open", "noted, not fixed", "known
  limitation") no longer satisfies the rule without a cited `BACKLOG` ID in the same entry. A
  targeted sweep for the same phrasing pattern turned up one more live instance of the identical gap:
  the coarse-tier evidence-upload short-circuit noted in decision #39's own Changelog entry above
  ("not logged as a numbered row... since it predates this pass's own scope") — logged now as
  `SAFE-019`. Also logged `GOV-020` (decisions 36–39 have no §21 decision-ledger-coverage rows, the
  same class of gap `GOV-019` already covers for decision 32) — found in passing, not fixed, per
  `GOV-014`'s own discipline about not scope-creeping into an unrequested fix.

  **Implemented and tested this pass**: 451 tests passing (up from 442 at session start), 22 files
  modified, 3 new (`profile/cached.py`, `tests/unit/test_profile_cached.py`,
  `scripts/retrofits/verify_profile_cache_against_aspose_page_foss.py` — kept after use per
  `GOVERNANCE.md` placement rule 5); `ruff check .`, `ruff format --check .`, `mypy src` all clean.
  Nothing was pushed to any product repo — the only live network operations this pass were read-only
  `git ls-remote`/`git clone --depth 1` against the real, public `aspose-page-foss` repo. User
  directive, developed via plan mode with iterative user correction on both the mode-gate scope and
  the `GOV-014` strengthening.
