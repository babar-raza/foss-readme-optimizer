**Clone-reliability redesign (`SCL-009`, new row).** The 300s timeout was a symptom: tracing one
`supervise_repo()` run found `clone_baseline()` is called independently by ~7-9 stateless
capabilities with no shared per-run repo view (decision #26(b)) — the same repo was cloned over
the network ~9-17 times per run, each an independent roll against `SCL-004`'s own measured
158s-1004s clone-time variance. This is the structural cause of cross-rerun inconsistency and a
latent correctness gap (nothing guaranteed all ~9 clones saw the same commit if upstream was
pushed to mid-run). Preserved unchanged: push-blocked clone model, `--depth 1`, decision #40's gate
split, the git-tree-API profiling path (`SCL-004`) as a named, not-this-pass, future
generalization. Fixed: an in-process memo in `clone.py` collapses the within-run redundancy to one
real clone; a new `invalidate_baseline_clone()`, called by `supervise_repo()` before its own
top-level clone, keeps every *call* (not merely every OS process) genuinely fresh —
`test_supervisor_loop.py`'s own consecutive-run tests proved a process-scoped-only memo was wrong.
`env.py::git_clone_timeout_seconds()` (default 300s→600s, evidence-informed, not a guarantee);
bounded transient-only retry (2 attempts, 5s/15s backoff; real errors fail fast); `supervise_repo()`
now catches its own clone's `GitSafetyError` and returns `BLOCKED` with written evidence instead of
an uncaught abort.
